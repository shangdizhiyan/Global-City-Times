# 全球时间对照工具开发文档

## 1. 项目定位

### 1.1 项目背景
外贸业务在客户开发、跟进、报价、会议预约、即时沟通中，经常需要快速判断：

- 某个国家或客户所在城市当前时间
- 与北京时间的时差
- 对方是否处于工作时间
- 多个目标国家在同一时刻的工作状态

通用世界时钟工具虽然很多，但通常不适合外贸场景，主要问题是：

- 只显示时间，不显示是否适合联系客户
- 缺少“北京时间对照”这一核心视角
- 不支持按国家、市场、客户分组管理
- 不能结合外贸常见工作时段做快速判断

因此需要开发一款面向外贸业务的“全球时间对照工具”。

### 1.2 产品目标
构建一个用于外贸客户开发与跟进的时间决策工具，帮助用户在最短时间内判断：

- 目标国家现在几点
- 与北京时间相差多少小时
- 当前是否属于工作时间
- 何时是最适合开发、跟进、打电话、发 WhatsApp 或邮件的时间段

## 2. 目标用户

- 外贸业务员
- 外贸主管
- 跨境销售团队
- 需要覆盖多个国家时区的国际业务人员

## 3. 核心使用场景

### 3.1 国家时间查询
用户输入国家名，例如“德国”“美国”“巴西”，系统显示：

- 当前本地时间
- 对应北京时间
- 时差
- 今天是否夏令时
- 当前是否工作时间

### 3.2 多国家对比
用户把重点开发国家加入列表，例如：

- 美国
- 加拿大
- 英国
- 德国
- 澳大利亚

系统在一个页面中统一展示这些国家当前时间及工作状态，方便决定今天优先联系谁。

### 3.3 最佳联系时间判断
系统基于目标国家工作时间，提示：

- 当前可联系
- 即将上班
- 即将下班
- 建议联系北京时间区间

### 3.4 客户跟进辅助
后续版本可支持给客户绑定国家/城市/时区，形成客户时间卡片，避免在错误时间联系客户。

## 4. 产品范围

### 4.1 MVP 第一阶段
首个版本建议只做以下功能：

- 支持国家搜索
- 支持北京时间与目标国家时间对照
- 支持多个国家收藏
- 支持显示工作时间状态
- 支持常见国家默认工作时间
- 支持夏令时自动处理

### 4.2 第二阶段

- 支持城市级时区
- 支持客户分组管理
- 支持“最佳联系时间推荐”
- 支持按地区筛选，如欧洲、北美、中东、东南亚
- 支持自定义工作时间

### 4.3 第三阶段

- 支持节假日避让
- 支持提醒功能
- 支持导出客户联系时间表
- 支持桌面通知
- 支持离线桌面版

## 5. 关键业务规则

## 5.1 时间换算规则

- 基准时间统一使用 IANA Time Zone 数据库，例如 `Asia/Shanghai`、`Europe/Berlin`、`America/New_York`
- 北京时间固定为 `Asia/Shanghai`
- 所有目标国家时间均由其标准时区实时换算
- 涉及夏令时时，必须由时区库自动处理，不能写死偏移量

### 5.2 国家与时区关系
这里必须明确一个重要事实：

- 有些国家只有一个主要时区，例如德国、日本
- 有些国家有多个时区，例如美国、加拿大、澳大利亚、俄罗斯、巴西

因此系统不能简单把“一个国家 = 一个时间”完全写死。建议采用下面策略：

- MVP：给每个国家配置一个“默认业务时区”
- 进阶版：允许用户在国家下选择具体城市或时区

示例：

- 美国默认：`America/New_York`
- 加拿大默认：`America/Toronto`
- 澳大利亚默认：`Australia/Sydney`

### 5.3 工作时间规则
“全球各个国家的工作时间”并不存在完全统一的官方标准，因此产品上应采用“默认业务规则 + 用户可调整”的方案。

建议默认规则：

- 标准工作日：周一到周五
- 默认工作时间：09:00-18:00
- 中东部分国家：可配置为周日到周四
- 特殊国家可预置不同时间段

建议字段：

- 工作日集合
- 上班时间
- 下班时间
- 午休时间（可选）
- 是否启用本地节假日过滤（后续版本）

### 5.4 联系状态判断
根据当前目标地时间，系统输出以下状态之一：

- 未上班
- 工作中
- 午休中
- 即将下班
- 已下班
- 非工作日

并额外输出建议：

- 适合发即时消息
- 适合邮件开发
- 不建议电话联系

## 8. 技术方案对比

## 8.1 方案一：Web 端

### 优点

- 开发速度快
- 部署和更新方便
- 手机、平板、电脑都能访问
- 团队成员可直接共享使用
- 后续可继续扩展为账号体系、云同步、客户管理

### 缺点

- 纯浏览器环境下本地系统集成能力有限
- 如果完全离线要求很强，需要额外做缓存或 `PWA`

## 8.2 方案二：Windows EXE 桌面程序

### 优点

- 更像本地工具，启动后即可使用
- 可做系统托盘、桌面提醒、开机启动
- 离线能力更强

### 缺点

- 安装、升级、分发更麻烦
- 后续如果你想给手机或其他同事用，不方便
- 多端扩展成本高
- 如果后面要做云端客户数据同步，最终仍然要补 Web 或服务端

## 8.3 结合你的业务场景的判断
你当前核心需求是：

- 看国家时间和北京时间对照
- 看国家工作时间
- 服务外贸开发场景

这类需求本质上更偏“查询工具 + 业务辅助工具”，不是强依赖 Windows 本地能力的软件。因此第一阶段更适合：

`先做 Web 端`

原因：

- 你可以在办公室电脑、家里电脑、手机浏览器随时打开
- 后期加国家库、工作时间规则、收藏国家、客户分组都更容易
- 先验证需求，成本最低
- 做成响应式页面后，基本等于同时拥有“电脑版 + 手机版”

## 8.4 最优落地建议
建议采用下面路线：

1. 第一阶段做 `Web` 版 MVP
2. 架构上保持前端和业务逻辑清晰分层
3. 如果后续确实需要桌面提醒、托盘常驻、离线使用，再把 Web 项目封装成 Windows 桌面程序

桌面封装可考虑：

- `Tauri`
- `Electron`

其中如果只是把 Web 应用打包成轻量级桌面工具，优先建议 `Tauri`。

## 9. 推荐技术栈

### 9.1 Web 方案推荐

- 前端：React + TypeScript + Vite
- UI：Tailwind CSS 或 Ant Design
- 时间处理：Luxon 或 Day.js + timezone 插件
- 数据存储：
  - MVP：本地 `localStorage`
  - 后续：后端 API + 数据库

### 9.2 为什么不建议一开始就做 EXE

- 你现在要先验证“这个工具到底怎么用最顺手”
- 早期需求变化一定很多
- Web 的试错成本最低
- 等交互和数据模型稳定后，再封装 EXE 更稳

## 12. 风险与注意事项

### 12.1 时间准确性风险

- 不能手工写死 UTC 偏移
- 必须依赖可靠时区库处理夏令时

### 12.2 工作时间并非全国统一

- 同一个国家不同行业、不同公司工作时间可能不同
- 产品中必须允许修改默认工作时间

### 12.3 多时区国家问题

- 美国、加拿大、澳大利亚等国家不能只给一个绝对时间结论
- 应明确默认时区来源，并允许用户切换

## 13. 最终结论

对你当前这个项目，建议：

- 先做 `Web` 端
- 目标是尽快做出一个可用的外贸时间对照工具
- 后续如果你需要桌面常驻、提醒、离线能力，再封装成 Windows `EXE`

一句话判断：

`现在先做 Web，更合适；EXE 适合作为后续增强版本，而不是第一版。`
