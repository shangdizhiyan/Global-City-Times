# 城市版改版设计开发文档

## 1. 改版目标

当前版本以“国家”为主，适合做全球时间总览，但不完全贴合外贸真实开发流程。

现实里的客户开发更常见的是：

- 先选目标国家
- 再聚焦重点城市
- 再按城市里的客户、渠道、社媒、广告平台去执行

所以新版本不建议做成“只有国家”，也不建议做成“所有城市平铺”。

更合理的产品结构是：

- 国家作为市场层
- 城市作为执行层
- 客户作为跟进层

一句话结论：

`推荐做成“国家 > 城市 > 客户”的三层结构。`

## 2. 为什么不能直接按全球城市平铺

如果直接按城市平铺，会出现几个问题：

- 城市数量太大，数据维护成本高
- 用户搜索和浏览会很乱
- 很多业务判断仍然要回到国家层，比如货币、宗教、国家代码、广告平台、社媒生态
- 同一国家下多个城市常常共享大量国家级信息，只是时间、产业、客户密度不同

所以不建议首页直接把全球城市全部平铺出来。

## 3. 推荐的产品结构

### 3.1 三层模型

第一层：国家

- 用来做市场归类
- 承载国家级资料
- 承载国家级社媒、广告、支付、消费习惯信息

第二层：城市

- 用来做时间判断和客户开发执行
- 承载城市时区、主要产业、客户密度、港口、展会、渠道偏好
- 是业务员实际点开的主要工作对象

第三层：客户

- 绑定到某个国家和某个城市
- 用来做跟进、提醒、备注、状态管理

### 3.2 页面主逻辑

首页不再以“全部国家卡片”为主，而改成：

- 我的重点城市
- 今日可联系城市
- 今日客户任务
- 国家筛选入口

国家页变成“目录页”，城市页变成“工作页”。

## 4. 新版信息架构

### 4.1 首页

首页建议展示 4 块：

- 北京时间主时钟
- 今日优先开发城市
- 我的重点国家
- 今日客户任务面板

首页目标不是“看全世界”，而是“今天先做谁”。

### 4.2 国家页

国家页建议展示：

- 国家基础资料
- 国家市场洞察
- 主流社媒和广告平台
- 重点城市列表
- 该国客户统计

国家页作用：

- 判断这个市场值不值得做
- 决定先做哪个城市

### 4.3 城市页

城市页是新版核心。

建议展示：

- 城市当前时间
- 与北京时间时差
- 当前是否工作中
- 最佳联系北京时间
- 城市所属国家
- 城市主要产业
- 主要港口 / 机场
- 主流沟通语言
- 客户开发渠道建议
- 该城市客户列表

城市页作用：

- 这是业务员真正点进去工作的页面

### 4.4 客户页/客户抽屉

客户层建议绑定：

- 国家
- 城市
- 联系人
- 公司名称
- 职位
- WhatsApp
- 邮箱
- LinkedIn
- 最近联系时间
- 跟进状态
- 下次联系建议时间

## 5. 数据层设计

## 5.1 国家数据

国家表继续保留，主要存：

- 国家代码
- 国家中文/英文名
- 所属大洲
- 默认业务时区
- 国家级市场信息
- 社媒和广告平台
- 国家下的重点城市列表

示例：

```json
{
  "code": "US",
  "nameZh": "美国",
  "nameEn": "United States",
  "continent": "North America",
  "defaultTimezone": "America/New_York",
  "mainSocialPlatforms": ["Facebook", "Instagram", "LinkedIn", "TikTok"],
  "mainAdPlatforms": ["Google Ads", "Meta Ads", "LinkedIn Ads", "TikTok Ads"],
  "cities": ["New York", "Los Angeles", "Chicago", "Houston", "Miami"]
}
```

## 5.2 城市数据

新增城市表，建议作为重点数据结构。

城市字段建议：

- `id`
- `countryCode`
- `cityName`
- `cityNameZh`
- `timezone`
- `isPrimary`
- `region`
- `population`
- `mainIndustries`
- `mainPort`
- `mainAirport`
- `language`
- `contactWindowBeijing`
- `businessHours`
- `channelNotes`

示例：

```json
{
  "id": "us-new-york",
  "countryCode": "US",
  "cityName": "New York",
  "cityNameZh": "纽约",
  "timezone": "America/New_York",
  "isPrimary": true,
  "population": "8.8M",
  "mainIndustries": ["Finance", "Retail", "Import trade", "Fashion"],
  "mainPort": "Port of New York and New Jersey",
  "mainAirport": "JFK",
  "language": "English",
  "businessHours": {
    "workdays": [1, 2, 3, 4, 5],
    "start": "09:00",
    "end": "18:00"
  }
}
```

## 5.3 客户数据

客户表新增强制城市绑定：

```json
{
  "id": "client-001",
  "name": "ABC Trading",
  "countryCode": "US",
  "cityId": "us-new-york",
  "contactPerson": "John",
  "status": "follow_up",
  "lastContactAt": "2026-04-09T09:20:00+08:00"
}
```

## 6. 产品交互建议

## 6.1 推荐搜索逻辑

搜索建议支持三种入口：

- 搜国家
- 搜城市
- 搜客户

搜索结果优先级建议：

1. 城市
2. 国家
3. 客户

因为未来新版最常用的是城市。

## 6.2 推荐导航结构

桌面端左侧导航建议改成：

- 仪表盘
- 国家市场
- 城市工作台
- 客户管理
- 今日任务
- 设置

## 6.3 国家与城市的展示关系

建议这样做，不要反过来：

- 国家页里展示城市列表
- 城市页里回看所属国家

不要做成：

- 城市主列表下再拼国家块

因为城市数量会很快变多，国家仍然应该是主目录。

## 7. MVP 改版范围

如果要尽快落地，我建议分三期。

### 第一期：城市化基础版

只做：

- 国家下面增加重点城市
- 每个城市有独立时间卡片
- 客户绑定城市
- 搜索支持城市
- 首页增加“今日优先城市”

这一期就能明显提升实用性。

### 第二期：城市工作台版

增加：

- 城市详情页
- 城市产业信息
- 主要港口/机场
- 城市渠道建议
- 城市客户清单

### 第三期：城市经营版

增加：

- 城市维度客户统计
- 城市维度成交率
- 城市维度提醒
- 城市维度最佳联系时间推荐
- 城市维度开发记录

## 8. 推荐页面方案

## 8.1 首页卡片

建议首页优先展示“城市卡片”，不是国家卡片。

城市卡片建议包含：

- 城市名
- 所属国家
- 当前时间
- 与北京时间时差
- 当前工作状态
- 今日建议联系窗口
- 已绑定客户数

## 8.2 国家页布局

国家页可做成：

- 顶部：国家概览
- 中部：市场洞察
- 下部：重点城市网格

## 8.3 城市页布局

城市页建议做成 5 个模块：

- 时间与联系判断
- 城市概况
- 渠道建议
- 客户列表
- 今日行动建议

## 9. EXE 客户端改造建议

你现在是做桌面端，所以这次改造非常适合往“工作台”走，而不是继续像网页信息站。

桌面端建议重点强化：

- 左侧固定导航
- 中间主工作区
- 右侧客户详情抽屉

也就是更像 CRM + 时间工作台，而不是纯卡片展示页。

## 10. 推荐开发顺序

建议按这个顺序改：

1. 新增 `cities` 数据层
2. 把客户改成绑定城市
3. 首页增加“重点城市”
4. 搜索支持城市
5. 国家详情页增加“重点城市列表”
6. 新建城市详情页
7. 再把 exe 首页从“国家卡片”重构成“城市工作台”

## 11. 结论

这次改版最合理的方向不是：

- 只做国家
- 也不是把全球所有城市直接平铺

而是：

`国家做市场层，城市做执行层，客户做跟进层`

这是最符合外贸实际的结构，也最适合做成一个真正能每天打开使用的 exe 客户端。

## 12. 最终建议

如果你让我直接推进开发，我建议下一步不要先乱改界面，而是先做这三件事：

1. 设计国家-城市-客户三层数据结构
2. 先为重点国家补重点城市
3. 先把首页改成“重点城市 + 今日任务”

这样改动最稳，也最容易一步一步落地。
